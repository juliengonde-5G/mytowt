## 2. AUDIT DE SECURITE

### 2.1 Points positifs
- CORS restreint aux domaines autorises (`https://my.towt.eu`, `http://51.178.59.174`)
- Security headers complets (CSP, HSTS, X-Frame-Options, X-Content-Type-Options, Referrer-Policy, Permissions-Policy)
- CSRF middleware actif (`app/csrf.py`)
- Maintenance mode middleware present
- SQL injection : `ALLOWED_TABLES` whitelist + `.bindparams()` dans admin_router
- Validation SECRET_KEY au demarrage (warning si valeur par defaut)
- Port DB (5433) expose sur toutes les interfaces mais app (8000) en 127.0.0.1 uniquement
- security.txt conforme a la RFC

### 2.2 Vulnerabilites detectees

#### CRITIQUE (a corriger immediatement)

| # | Probleme | Fichier | Impact |
|---|----------|---------|--------|
| S1 | **SECRET_KEY par defaut en dur dans config.py** | `app/config.py:17` | Si `.env` absent, la cle `towt_secret_key_change_in_production_2025` est utilisee. Permet la forge de tokens de session. |
| S2 | **Mot de passe DB par defaut en dur** | `app/config.py:14`, `docker-compose.yml:8,27` | `towt_secure_2025` en fallback. Si `.env` oublie, la DB est accessible avec des credentials publics. |
| S3 | **Port PostgreSQL expose sur 0.0.0.0:5433** | `docker-compose.yml:33` | La DB est accessible depuis l'exterieur du serveur. Devrait etre `127.0.0.1:5433:5432`. |
| S4 | **Cookie session sans flags securises** | `app/auth.py` | Le cookie `towt_session` est set sans `httponly`, `secure`, ni `samesite`. Vulnerable au vol XSS et CSRF. |

#### HAUTE

| # | Probleme | Fichier | Impact |
|---|----------|---------|--------|
| S5 | **onboard_router: permissions manquantes sur 20+ routes** | `app/routers/onboard_router.py` | Routes ATA/ATD, SOF, closure, upload utilisent `get_current_user` au lieu de `require_permission("captain", "M")`. Tout utilisateur authentifie peut enregistrer des operations de bord. |
| S6 | **stowage_router et tracking_router sans auth au niveau router** | `app/main.py:207-209` | `stowage_router` et `tracking_router` inclus sans `dependencies=[Depends(require_permission(...))]`. |
| S7 | **CSP autorise `unsafe-inline` pour scripts et styles** | `app/main.py:110` | Atenue la protection XSS. |

#### MOYENNE

| # | Probleme | Fichier | Impact |
|---|----------|---------|--------|
| S8 | `except:` nu (bare except) | `crew_router.py:33`, `pricing_router.py`, `claim_router.py:93` | Masque les erreurs systeme (KeyboardInterrupt, SystemExit). |
| S9 | Session token sans expiration courte | `app/config.py:19` | 8h d'expiration. Acceptable pour usage interne mais a surveiller. |
| S10 | Pas de rate limiting sur `/login` | `app/routers/auth_router.py` | Bruteforce possible. |

#### BASSE

| # | Probleme | Fichier | Impact |
|---|----------|---------|--------|

| S12 | Pas d'Alembic pour les migrations | Projet | Risque de drift schema. |

## 4. SAUVEGARDE BASE DE DONNEES

- Script cree : `scripts/backup_db.sh`
- Format : `pg_dump` compresse (gzip)
- Rotation : suppression automatique apres 30 jours (configurable via `BACKUP_RETENTION_DAYS`)
- Sortie : `backups/towt_YYYYMMDD_HHMMSS.sql.gz`
- **Note** : le script necessite Docker en fonctionnement (`towt-db` container). Non executable dans cet environnement de dev.

**Recommandation** : configurer un cron quotidien :
```
0 2 * * * /path/to/mytowt/scripts/backup_db.sh >> /var/log/towt-backup.log 2>&1
```
